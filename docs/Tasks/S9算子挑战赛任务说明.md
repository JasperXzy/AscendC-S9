# 昇腾AI创新大赛-算子挑战赛（S9赛季）任务说明

本文根据本仓库中的 `docs/S9挑战性能赛题.xlsx`、`case_910b/` 公开测试样例，以及本地 CANN Community Edition 8.5.0 文档整理。

## 一、比赛任务

本赛季是 Ascend C 自定义算子性能挑战。任务是为 5 个 PyTorch 语义等价算子实现 Ascend C 自定义算子，并在精度验证通过的前提下优化性能。

公开样例位于：

- `case_910b/Concat`
- `case_910b/Greater`
- `case_910b/IndexAdd`
- `case_910b/SquareSumV1`
- `case_910b/Transpose`

赛题规格表位于：

- `docs/S9挑战性能赛题.xlsx`
- `docs/Tasks/S9挑战性能赛题.xlsx`

## 二、题目列表

| 序号 | 算子 | 参考语义 | 输入/输出概要 | 关键要求 |
|---:|---|---|---|---|
| 1 | `Concat` | `torch.cat(inputs, dim)` | `inputs: tensor_list`，`dim: int`，`output: tensor` | 支持 `float32/float16/int32/int8`，ND 格式；输出 shape 由 `dim` 决定；各维可能不是 32 的整数倍。 |
| 2 | `Greater` | `torch.gt(self, other)` | `self, other: tensor`，`output: bool tensor` | 支持 `float32/bfloat16/float16/int32/int8`；`self` 和 `other` 同类型；需要支持广播；需要正确处理 `inf`、`-inf`、`nan`。 |
| 3 | `IndexAdd` | `torch.index_add(self, dim, index, source)` | `self: tensor`，`index: int32 tensor(M)`，`source: tensor`，`dim: int` | 支持 `float32/bfloat16/float16/int32/int8`；`M in [1, 8000]`；`source` 除 `dim` 维长度等于 `index.length` 外，其他维度与 `self` 对应维度一致。 |
| 4 | `Transpose` | `torch.permute(input, dims)` | `input: tensor`，`dims: list_int`，`output: tensor` | 支持 `float32/float16/int32/int8`，ND 格式；多维转置，各维可能不是 32 的整数倍。 |
| 5 | `SquareSumV1` | `torch.sum(input ** 2, dim=axis, keepdim=keep_dims)` | `input: tensor`，`axis: list_int`，`keep_dims: bool`，`output: tensor` | 支持 `float16/bfloat16/float`；输出 shape 由 `axis` 和 `keep_dims` 决定；各维可能不是 32 的整数倍。 |

通用 shape 范围来自赛题表：

- 多数题目维度形如 `(..., N4, N3, N2, N)`。
- `N in [1, 10000]`
- `N2 in [1, 10000]`
- `N3 in [1, 1000]`
- `N4 in [1, 1000]`
- `Transpose` 额外包含 `N5 in [1, 1000]`
- `SquareSumV1` 中 `N4 in [1, 200]`
- `IndexAdd` 中 `M in [1, 8000]`

注意：公开 `test_op.py` 只覆盖很少的样例 shape，正式评测很可能使用隐藏用例，因此实现不能只写死公开样例。

## 三、CANN/Ascend C 开发流程

CANN 工程化 Ascend C 算子开发的核心流程如下：

1. 准备 CANN、驱动固件、昇腾设备运行环境。
2. 用 `msopgen` 基于算子原型 json 创建 Ascend C 算子工程，或基于官方样例工程改造。
3. 在 `op_host/` 中实现 host 侧逻辑，包括算子原型注册、shape 推导、信息库、tiling。
4. 在 `op_kernel/` 中实现 AI Core kernel。
5. 执行 `./build.sh` 编译算子工程，生成自定义算子安装包 `custom_opp_<target-os>_<target-arch>.run`。
6. 安装 `.run` 包并配置环境变量，使自定义算子在测试环境中生效。
7. 使用 `case_910b/<Op>/run.sh` 或自建测试验证精度和性能。
8. 使用主办方提供的 `zip_op.sh` 打包最终提交文件。

CANN 文档中的标准工程结构示例：

```text
AddCustom
├── build.sh
├── cmake
├── CMakeLists.txt
├── CMakePresets.json
├── framework
├── op_host
│   ├── add_custom_tiling.h
│   ├── add_custom.cpp
│   └── CMakeLists.txt
├── op_kernel
│   ├── CMakeLists.txt
│   └── add_custom.cpp
└── scripts
```

对本比赛最重要的是：

- `op_host/`：放 host 侧实现，负责原型、shape 推导、tiling。
- `op_kernel/`：放 kernel 侧实现，负责 AI Core 上实际计算。
- kernel 侧依赖文件必须放在 `op_kernel/` 目录内，否则源码发布/在线编译场景可能找不到依赖。
- `CMakePresets.json` 或 `cmake/config.cmake` 中的 `ASCEND_COMPUTE_UNIT`、CANN 路径、编译模式等需要与运行环境匹配。

## 四、本仓库公开样例如何测试

每道题目录里都有类似结构：

```text
case_910b/<Op>/
├── common/pytorch_npu_helper.hpp
├── extension/custom_op.cpp
├── get_time.py
├── run.sh
├── setup.py
└── test_op.py
```

测试入口示例：

```bash
cd case_910b/Concat
bash run.sh 1
```

`run.sh` 的逻辑大致是：

1. 首次执行或需要重装时，运行 `python3 setup.py build bdist_wheel` 生成 pybind wheel。
2. 安装 `dist/custom_ops*.whl`。
3. 用 `msprof --application="python3 test_op.py <case_id>"` 跑测试。
4. 通过 `get_time.py` 读取 profiling 耗时。
5. 同时检查精度和性能。

注意：`extension/custom_op.cpp` 是 PyTorch/pybind 调用封装层；比赛核心提交仍是 Ascend C 自定义算子源码目录和编译出的 `.run` 包。

## 五、最后要提交什么

最终提交的是主办方脚本生成的 `.zip` 文件。

截图中的提交规则要求：

1. 将源码文件 `op_host/`、`op_kernel/` 和编译生成的 `custom_*.run` 放在同一个文件夹中。
2. 只支持打包成 `.zip` 格式提交。
3. 必须使用主办方提供的 `zip_op.sh` 打包脚本，否则成绩无效。
4. 提交源码必须是最后一次上榜时使用的代码。
5. 如果 kernel 代码与 `.run` 包不一致，或算子性能与榜上性能不一致，本次成绩无效。

建议最终目录结构：

```text
<OpName>_zip/
├── custom_opp_<target-os>_<target-arch>.run
├── op_host/
│   ├── CMakeLists.txt
│   ├── <op>.cpp
│   └── <op>_tiling.h
└── op_kernel/
    ├── CMakeLists.txt
    └── <op>.cpp
```

打包命令按主办方脚本为准，截图示例形如：

```bash
bash zip_op.sh AddCustom
```

产物形如：

```text
AddCustom.zip
```

对于本赛季 5 道题，应分别生成对应算子的提交 zip，除非平台另有合并提交要求。

## 六、注意事项

1. 不要只针对公开样例写死 shape。正式评测会覆盖更多 dtype、shape、非对齐长度和特殊场景。
2. 非 32 对齐是通用要求，尾块处理必须正确。
3. `Greater` 需要处理广播和 `nan` 比较。PyTorch 中涉及 `nan` 的 `>` 比较通常为 `False`。
4. `IndexAdd` 需要考虑同一个 index 重复出现时的累加语义。
5. `SquareSumV1` 的累加精度要与 PyTorch 容差匹配，公开样例对 fp16 使用相对宽松但仍需要控制误差。
6. `Transpose` 需要正确处理任意 `dims` 排列，不应只覆盖二维 `(1, 0)`。
7. 编译出的 `.run` 包要与提交源码一一对应，提交前不要再改 `op_host/` 或 `op_kernel/`。
8. 如果使用指定目录安装 `.run` 包，需要执行生成的 `set_env.bash`，让 `ASCEND_CUSTOM_OPP_PATH` 生效。
9. 多个自定义算子包共存时，要注意 `vendor_name` 和加载优先级，避免同名 OpType 被其他包覆盖。
10. 提交 zip 内不要放无关文件，尤其不要把 `dist/`、profiling 结果、缓存、临时日志混进去。
